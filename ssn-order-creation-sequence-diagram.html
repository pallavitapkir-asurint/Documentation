asurint-api — authLink JWT Re-signing (Critical)

The Okta JWT is replaced with an internally-signed JWT containing full UserPrincipal

16
asurint-api authLink.ts — JWT re-signing ⭐ KEY STEP
This is the critical security bridge. The Okta JWT from the browser is discarded. A new JWT is minted using the internal REMOTE_SCHEMA_JWT_SECRET and signed with the full UserPrincipal as the payload. This new JWT is forwarded to reports-api.
asurint-apiasurint-api/src/apollo/schemas/remote/links/authLink.ts
// authLink.ts — JWT re-signing const jwtUtil = new JWT(getConfiguration().remoteSchemaJwtSecret); // Sign new JWT with full UserPrincipal as payload const newToken = await jwtUtil.sign( authentication, // UserPrincipal object authentication.emailAddress // subject claim ); // Replace Authorization header on outgoing request headers["Authorization"] = `Bearer ${newToken}`;
  • Why: reports-api (Kotlin/Spring Boot) cannot validate Okta JWTs. It validates using the shared REMOTE_SCHEMA_JWT_SECRET instead.
  • Payload: Full UserPrincipal — emailAddress, personas, userId, orgUnitId — embedded in JWT claims
  • Secret source: REMOTE_SCHEMA_JWT_SECRET env var shared between asurint-api and all downstream Kotlin services
  • Effect: reports-api receives a trusted internal JWT it can verify without Okta dependency
⭐ The browser never knows about this JWT swap. reports-api trusts asurint-api as its sole gateway.
